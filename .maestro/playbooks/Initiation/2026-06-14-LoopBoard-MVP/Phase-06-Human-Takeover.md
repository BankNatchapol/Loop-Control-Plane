# Phase 06: Human Takeover

This phase implements the control handoff between AI-owned work and human manual editing. The user should be able to claim an AI task, open the right local workspace, copy a Claude Code prompt, update handoff context, and return the task to AI when ready.

## Tasks

- [x] Inspect existing task actions, GitHub labels, context files, and local open helpers before extending takeover behavior:
  - Reuse current event creation, handoff generation, GitHub label sync, and VS Code open patterns
  - Keep takeover state changes atomic from the user's perspective
  - Preserve all security constraints around shell execution and token redaction
  - Completion notes (2026-06-15):
    - Existing task action flow is centered on `applyTaskAction` in `lib/loopboard.ts`, persisted by `LoopBoardRepository.applyTaskAction` in `lib/db/loopboard-repository.ts`, and exposed by `app/api/tasks/[taskId]/actions/route.ts`. Repository writes already group the task row update and inserted task events in a SQLite transaction, then the route refreshes an existing `events.jsonl` through `syncExistingTaskEventsFile`.
    - Current action vocabulary includes `claim-human` and `return-ai`, but the event type names are currently `HUMAN_CLAIMED` and `RETURNED_TO_AI`; the later full takeover task should add or rename to the requested `HUMAN_TAKEOVER` and `ASSIGNED_TO_HUMAN` semantics deliberately instead of duplicating one-off event logic in the UI.
    - GitHub label behavior is centralized in `lib/github/github-issues.ts`, `lib/github/github-connection.ts`, and `app/api/tasks/[taskId]/github/labels/route.ts`. `calculateGitHubIssueLabels` already emits `human-working` for human-owned or `human-working` tasks and emits `ao-ready` only for low-risk ready tasks; label sync goes through the GitHub labels API and redacts token material in API error messages.
    - Task context files are generated by `TaskContextService` in `lib/context/task-context-service.ts`. It owns `task.md`, `context.md`, `handoff.md`, and `events.jsonl`; handoff refresh preserves the `LOOPBOARD:HUMAN_NOTES` region, and events export serializes the persisted event stream as JSONL.
    - Local open support is currently project-scoped in `lib/projects/project-open-actions.ts` and `app/api/projects/[projectId]/open/route.ts`. It validates `repoPath`, launches with `spawn(..., { shell: false })`, checks `code --version` for VS Code, and returns actionable missing-path or missing-CLI errors. There is not yet a task/worktree-specific open endpoint.
    - UI entry points already exist in `app/page.tsx` for task action buttons, GitHub label buttons, context file export/refresh, copy context paths, and project open actions. Future takeover work should extend those surfaces rather than introducing parallel state or shell execution paths.
